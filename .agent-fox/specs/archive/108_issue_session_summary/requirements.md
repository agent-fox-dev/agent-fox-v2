# Requirements Document

## Introduction

This specification adds an automated feedback loop from spec implementation
back to the originating GitHub issue. When all task groups of a spec are
fully implemented and merged to `develop`, the orchestrator posts a roll-up
summary comment to the issue referenced in the spec's `prd.md` `## Source`
section.

## Glossary

- **Source URL**: The issue tracker URL recorded in the `## Source` section
  of a spec's `prd.md`. Currently only GitHub issue URLs are supported.
- **Spec completion**: The state when all non-optional task groups in a
  spec's `tasks.md` have checkbox state `[x]`.
- **Roll-up summary**: A comment posted to the originating issue that
  summarizes the delivered implementation and references the merge commit.
- **Forge**: A source code hosting and issue tracking platform (e.g.,
  GitHub, GitLab, Linear).
- **Platform**: An instance of `PlatformProtocol` that provides issue
  tracking operations.

## Requirements

### Requirement 1: Source URL Extraction

**User Story:** As an operator, I want the system to extract the originating
issue URL from a spec's `prd.md` so that it knows where to post the
completion summary.

#### Acceptance Criteria

[108-REQ-1.1] WHEN a spec's `prd.md` contains a `## Source` section with a
line matching `Source: <url>`, THE system SHALL extract the URL and return
it to the caller.

[108-REQ-1.2] WHEN the extracted URL matches the pattern
`https://github.com/{owner}/{repo}/issues/{number}`, THE system SHALL
parse and return the owner, repo, and issue number as structured data.

[108-REQ-1.3] THE source URL parser SHALL be a pure function that accepts
a URL string and returns a structured result containing the forge type
and issue identifier, or `None` if the URL does not match any known
pattern.

#### Edge Cases

[108-REQ-1.E1] IF the `## Source` section is missing from `prd.md`, THEN
THE system SHALL return `None` without raising an exception.

[108-REQ-1.E2] IF the `Source:` line contains a non-URL value (e.g.,
`Source: Input provided by user via interactive prompt`), THEN THE system
SHALL return `None` without raising an exception.

[108-REQ-1.E3] IF `prd.md` does not exist in the spec directory, THEN
THE system SHALL return `None` without raising an exception.

### Requirement 2: Spec Completion Detection

**User Story:** As an operator, I want the system to detect when all task
groups of a spec are complete so that the summary is posted at the right
time.

#### Acceptance Criteria

[108-REQ-2.1] WHEN the orchestrator finishes dispatching all tasks, THE
system SHALL identify specs that are newly completed in the current run
by comparing `GraphSync.completed_spec_names()` against a set of specs
that have already had their issue summary posted.

[108-REQ-2.2] WHEN a spec is newly completed, THE system SHALL check
whether it has an originating issue URL by calling the source URL parser
on its `prd.md`.

#### Edge Cases

[108-REQ-2.E1] IF a spec was already completed in a previous run (and
the summary was already posted), THEN THE system SHALL NOT post a
duplicate summary comment.

### Requirement 3: Summary Comment Construction

**User Story:** As an issue author, I want to see a clear summary of
what was delivered and which commit it landed in, so that I can verify
the implementation without searching through branches.

#### Acceptance Criteria

[108-REQ-3.1] WHEN constructing the summary comment, THE system SHALL
include the spec folder name (e.g., `108_issue_session_summary`).

[108-REQ-3.2] WHEN constructing the summary comment, THE system SHALL
include the SHA of the `develop` branch HEAD at the time the summary
is generated.

[108-REQ-3.3] WHEN constructing the summary comment, THE system SHALL
include a list of task group titles derived from `tasks.md`.

[108-REQ-3.4] THE summary comment SHALL end with a footer line
`*Auto-generated by agent-fox.*` to distinguish it from human comments.

### Requirement 4: Comment Posting

**User Story:** As an issue author, I want the completion summary to
appear automatically on my issue so that I am notified without manual
cross-referencing.

#### Acceptance Criteria

[108-REQ-4.1] WHEN a spec is newly completed and has a valid GitHub
source URL, THE system SHALL call `PlatformProtocol.add_issue_comment()`
with the issue number and the constructed summary comment body.

[108-REQ-4.2] THE comment posting SHALL occur during the orchestrator's
end-of-run cleanup phase, after all sessions have finished.

#### Edge Cases

[108-REQ-4.E1] IF `add_issue_comment()` raises an exception, THEN THE
system SHALL log a warning and continue without affecting the run status.

[108-REQ-4.E2] IF the source URL's forge type does not match the
configured platform type, THEN THE system SHALL skip posting and log
an info-level message.

### Requirement 5: Platform Wiring

**User Story:** As an operator, I want the platform to be available to
the orchestrator so that it can post issue comments when specs complete.

#### Acceptance Criteria

[108-REQ-5.1] WHEN the orchestrator infrastructure is set up, THE system
SHALL create a `PlatformProtocol` instance using the existing platform
factory and pass it to the `Orchestrator` constructor.

[108-REQ-5.2] THE `Orchestrator` constructor SHALL accept an optional
`platform` parameter of type `PlatformProtocol | None`.

[108-REQ-5.3] THE platform factory SHALL return `None` instead of
calling `sys.exit(1)` when the platform type is `"none"` or when
required environment variables (e.g., `GITHUB_PAT`) are missing.

#### Edge Cases

[108-REQ-5.E1] IF the platform is `None` (not configured), THEN THE
system SHALL skip all issue summary operations silently without logging
any warnings.

### Requirement 6: Develop HEAD Commit SHA

**User Story:** As an issue author, I want the summary to reference the
exact commit so that I can inspect the delivered code.

#### Acceptance Criteria

[108-REQ-6.1] WHEN generating the summary, THE system SHALL read the
current `develop` branch HEAD SHA using `git rev-parse develop` and
include it in the comment.

#### Edge Cases

[108-REQ-6.E1] IF `git rev-parse develop` fails, THEN THE system SHALL
use the string `"unknown"` as the commit SHA and still post the comment.
