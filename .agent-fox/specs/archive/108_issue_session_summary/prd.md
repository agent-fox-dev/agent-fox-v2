# PRD: Issue Session Summary

## Problem

When a spec originates from a GitHub issue (via `af-spec`), the `## Source`
section in `prd.md` records the issue URL. However, once the spec is fully
implemented — all task groups completed and merged to `develop` — there is no
feedback loop to the originating issue. The issue author has no visibility
into whether implementation landed, which commit it shipped in, or what
was delivered. They must manually cross-reference branches and commits to
confirm completion.

## Goal

After all task groups of a spec are implemented and merged to `develop`, the
orchestrator automatically posts a roll-up summary comment to the originating
GitHub issue. The comment references the final merge commit and summarizes
what was delivered. This closes the issue-to-code feedback loop without
requiring manual intervention.

## Scope

### In Scope

- Parsing the `## Source` section of `prd.md` to extract the originating
  issue URL.
- Detecting when all task groups of a spec are complete (all non-optional
  groups have `[x]` checkboxes in `tasks.md`).
- Constructing a roll-up summary comment with the develop merge commit SHA.
- Posting the comment to the originating issue via `PlatformProtocol`.
- Wiring `PlatformProtocol` through the orchestrator so it is available at
  spec-completion time.
- Best-effort URL parsing: support GitHub issue URLs now, with an extensible
  pattern for GitLab, Linear, or other forges in the future.
- Graceful degradation: skip silently if no platform is configured; warn and
  continue if comment posting fails.

### Out of Scope

- Closing the issue (not the agent's responsibility).
- Per-session progress comments (only a final roll-up when all groups are done).
- Per-task-group progress updates.
- Creating or modifying `PlatformProtocol` method signatures (the existing
  `add_issue_comment()` method is sufficient).
- Supporting non-issue source types (e.g., file paths, user prompts) —
  these have no issue to comment on.

## Feature Details

### Source URL Parsing

The `## Source` section in `prd.md` contains a line like:

```
Source: https://github.com/owner/repo/issues/42
```

The system extracts the URL using a best-effort regex. For GitHub, this
matches `https://github.com/{owner}/{repo}/issues/{number}`. The URL is
not validated against the platform config — it is taken at face value.

If the `## Source` section is missing, contains a non-URL value (e.g.,
`Source: Input provided by user via interactive prompt`), or the URL does
not match a known issue URL pattern, the system skips silently.

### Completion Detection

The orchestrator already tracks spec completion via
`GraphSync.completed_spec_names()`. When a spec enters the completed set
for the first time during a run, the system checks whether it has an
originating issue URL. This detection happens in the orchestrator's
end-of-run cleanup phase, alongside existing post-completion tasks
(audit cleanup, consolidation, memory rendering).

### Summary Comment

The comment posted to the issue follows this template:

```markdown
## Spec Implemented

Spec `{spec_name}` has been fully implemented and merged to `develop`.

**Commit:** `{commit_sha}`

### Summary

{roll_up_summary}

---
*Auto-generated by agent-fox.*
```

The roll-up summary is derived from the task group titles in `tasks.md`.
The commit SHA is the HEAD of `develop` at the time the last task group
was completed and merged.

### Platform Wiring

The `PlatformProtocol` instance is already created by
`nightshift/platform_factory.py` for the nightshift daemon. The same
factory is reused to provide a platform instance to the orchestrator.

The orchestrator receives an optional `platform` parameter. If
`platform` is `None` (no platform configured), the issue summary feature
is silently disabled. If a platform is provided but `add_issue_comment()`
fails (network error, auth failure, etc.), the error is logged as a
warning and the run continues normally.

### Extensibility

The source URL parser is designed as a simple function that takes a URL
string and returns a structured result (forge type + issue number) or
`None`. Adding support for a new forge (e.g., GitLab
`https://gitlab.com/group/project/-/issues/42`) requires only adding a
new regex pattern to the parser — no architectural changes.

## Design Decisions

1. **Final roll-up only, not per-session:** Posting per-session or
   per-task-group updates would create noise on the issue. A single
   summary when all groups are done is cleaner and more useful.

2. **Best-effort URL parsing:** The source URL format is not validated
   against the configured platform. This allows the system to work even
   if the platform config doesn't match the source URL forge (e.g., a
   GitHub issue URL in a project that hasn't configured the platform).

3. **Skip silently when no platform:** If `platform.type == "none"` or
   no `GITHUB_PAT` is available, the feature does nothing. No error, no
   warning — the session is still successful.

4. **Warn on comment failure:** If the platform is configured but the
   comment post fails, a warning is logged. The session and run are still
   considered successful.

5. **Source URL is forge-agnostic:** The parser returns a structured
   result with the forge type, so the same mechanism can support GitHub,
   GitLab, Linear, or other forges in the future without changing the
   orchestrator logic.

6. **Reuse existing `create_platform` factory:** Rather than inventing a
   new platform instantiation path, the orchestrator uses the same
   `create_platform()` from `nightshift/platform_factory.py`, adapted to
   return `None` instead of calling `sys.exit(1)` when the platform is
   not configured.

7. **Commit SHA from `develop` HEAD:** The commit referenced in the
   summary is the HEAD of `develop` at the time the spec-completion
   hook runs, which is the merge commit from the final task group's
   harvest.
