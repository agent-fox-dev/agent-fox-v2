## Gotchas

- Never remove error handling, logging, or 'why' comments during refactoring; these are load-bearing even if they appear cluttered. Only delete comments that restate what code does. _(spec: 103_remove_hook_runner, confidence: 0.90)_
- For autonomous code fixing, check for a dirty working tree before proceeding and require the developer to commit or stash uncommitted changes to avoid accidental data loss. _(spec: 103_remove_hook_runner, confidence: 0.90)_
- When both bash_allowlist and bash_allowlist_extend are configured, bash_allowlist takes precedence and bash_allowlist_extend should be ignored with a warning logged. _(spec: 103_remove_hook_runner, confidence: 0.90)_
- Configuration generation helpers (bounds maps, descriptions) must be kept in sync with actual config classes; removing a config field requires removing corresponding entries from these maps. _(spec: 103_remove_hook_runner, confidence: 0.90)_
- Hypothesis property tests with `st.text()` strategy may generate whitespace-only strings; use `.filter(lambda s: s.strip())` to avoid meaningless test cases. _(spec: 103_remove_hook_runner, confidence: 0.90)_
- Floating-point cost accumulation in aggregates should use approximate equality checks (e.g., `abs(actual - expected) < 1e-4`) rather than exact equality to avoid precision errors. _(spec: 105_db_plan_state, confidence: 0.90)_
- Session outcome records with `error_message=None` must store SQL NULL in the database, not an empty string, to distinguish successful completions from failures. _(spec: 105_db_plan_state, confidence: 0.90)_
- Property-based tests with Hypothesis should use unique identifiers (UUIDs) per iteration to prevent file accumulation and path conflicts in tmp_path fixtures. _(spec: 105_db_plan_state, confidence: 0.90)_
- When using Hypothesis with pytest fixtures (like tmp_path), suppress the function_scoped_fixture health check to avoid warnings during property-based testing. _(spec: 103_agent_conversation_trace, confidence: 0.90)_
- Migration version numbers in task documentation may become outdated when earlier migrations are added after planning; use the actual implemented version number in code comments and track it separately from spec documentation. _(spec: 105_db_plan_state, confidence: 0.90)_
- Migration version numbers may skip versions in documentation if a newer migration landed before the documented one was implemented; update comments to reflect actual migration versions applied. _(spec: 104_adaptive_retrieval, confidence: 0.60)_
- DuckDB 1.5.1 does not allow a read_only=True connection and a write connection to the same file in the same OS process simultaneously; they must be opened sequentially or in separate processes. _(spec: 105_db_plan_state, confidence: 0.90)_
- Fact IDs from multiple sources (DuckDB, causal traversal, entity lookup) are cast to strings and stored in ScoredFact dataclass; content strings are coerced to empty string if None. _(spec: 104_adaptive_retrieval, confidence: 0.90)_
- Topological sort initialization uses a deque with newly-ready facts inserted in score-descending order, but the queue is processed FIFO, so lower-scoring facts may be processed before higher-scoring ones added later. _(spec: 104_adaptive_retrieval, confidence: 0.60)_
