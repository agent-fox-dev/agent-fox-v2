Comparative Analysis: Agent-Fox Versions 7, 8 & 9

  Context

  All three branches were generated from the same spec corpus (9 component specifications in .agent-fox/specs/), defining an SDV (Software-Defined Vehicle) parking payment system with Rust RHIVOS services, Go cloud backends, mock apps, and infrastructure (NATS, Kuksa Databroker).

  - version_7 — agent-fox 3.3.1, parallel=2, max_budget=$8
  - version_8 — agent-fox 3.1.0, parallel=5, max_budget=$8
  - version_9 — agent-fox 3.1.0, parallel=5, max_budget=$15 (longer session timeout, more config options)

  Confusingly, version_7 was generated by a newer agent-fox release (3.3.1) than versions 8 and 9 (both 3.1.0). The version numbers refer to the branch iteration, not the tool version.

  ---
  What Changed Version to Version

  v7 -> v8: Robustness and breadth

  1. Signal handler race fix — v8 introduced a watch channel pattern for shutdown signals, preventing signal loss during command processing. v7 used a simpler tokio::select! that could miss a SIGTERM delivered while processing a lock command.
  2. More errata — v8 produced 20 errata documents capturing API migrations (kuksa v1->v2), spec contradictions, race guard patterns, and proto ambiguities. v7 produced none. This reflects v8's deeper engagement with spec-vs-reality gaps.
  3. Expanded Go services — Go source LOC tripled (1,908 -> 6,429). v8 filled out the cloud-gateway with proper auth, natsclient, and store subpackages, whereas v7 had a thinner implementation.
  4. Lifecycle test — v8 added lifecycle_test.go for the locking-service (startup/shutdown verification), missing in v7.
  5. TOCTOU-safe state transitions — v8's update-service uses transition_from(expected_state, new_state) to prevent time-of-check/time-of-use races. v7's state management was simpler.
  6. More test exclusions documented — Both versions exclude cloud-gateway-client and update-service from make test (they contain stub tests from unfinished specs), but v8 documented why in docs/errata/01_test_scope.md.

  v8 -> v9: Test density and conciseness

  1. Rust test LOC nearly doubled (2,213 -> 4,282) while Rust source LOC decreased (8,527 -> 6,098). v9 wrote leaner production code with more thorough test coverage — a clear quality improvement.
  2. Fewer errata (20 -> 6). v9 either avoided the spec ambiguities v8 stumbled on, or resolved them inline rather than documenting workarounds. The remaining 6 errata are focused and actionable.
  3. Cleaner module visibility — v9 uses mod (private) instead of pub mod for internal modules like broker, command, config. v7 and v8 exported everything publicly. This is better Rust practice.
  4. More direct signal handling — v9 registers SIGTERM directly with tokio::signal::unix::signal() instead of v8's spawned watch-channel pattern. Simpler, fewer moving parts, same correctness.
  5. Makefile improvements — v9 uses variables (CARGO_TEST_EXCLUDE, GO_MODULES) and iterates over module lists. v8 hardcoded each cd command. v9's approach is more maintainable.
  6. Expanded Go test infrastructure — v9 added overlay_test.go for databroker VSS overlay verification, parking_app_grpc_test.go for direct gRPC testing, and embedded kuksav2 proto stubs in the test directory for self-contained integration tests.
  7. Slightly worse unwrap() hygiene — v9 has 191 unwrap() calls vs v8's 173 (and v7's 134). However, v9 reduced expect() calls from 122 to 93, suggesting a shift toward unwrap() in test code where panics are acceptable, and better error handling in production paths.

  ---
  What Improved

  1. Test-to-code ratio is the clearest improvement trajectory:
    - v7: ~28% of Rust LOC is tests
    - v8: ~26% of Rust LOC is tests
    - v9: 70% of Rust LOC is tests (4,282 test / 6,098 source)

  v9 achieved more with less code, and tested it harder.
  2. Errata quality over quantity — v8's 20 errata suggest the agent struggled with spec ambiguities and over-documented workarounds. v9's 6 errata are precise (API migration decisions, test scope). This suggests agent-fox learned to handle ambiguity more decisively.
  3. Code organization matured — private modules in v9, Makefile variables, self-contained test fixtures with embedded proto stubs.
  4. Go backend completeness grew steadily (1.9K -> 6.4K -> 7.8K LOC), with each version adding more depth to auth, NATS client, and handler implementations.

  What Did NOT Improve

  1. Same test exclusions persist — All three versions exclude cloud-gateway-client and update-service from the test suite due to unfinished stub tests. This is a persistent blind spot: agent-fox generates stub tests as part of spec scaffolding but never goes back to implement them.
  2. unwrap() count keeps rising (134 -> 173 -> 191). While many are likely in test code (where panics are fine), it shows agent-fox doesn't actively minimize unwrap() in production code over iterations.
  3. No integration tests spanning multiple services — All three versions test services in isolation. There's no end-to-end test that starts the locking-service, databroker, and cloud-gateway together and runs a full lock/unlock cycle. Each version adds more unit and smoke tests but doesn't close the integration gap.
  4. No persistent state — All three versions use in-memory state. This is per-spec (demo scope), but it's notable that no version attempted to challenge this limitation.

  ---
  Qualitative Verdict

  There is a meaningful qualitative difference across the three versions, though it's not a simple linear improvement:

  - v7 produced a solid, complete implementation of specs 01-08 with good code quality, but with less attention to edge cases, race conditions, and spec-reality gaps. It's the "naive first pass" — correct but not hardened.
  - v8 went deeper into robustness: race guards, TOCTOU-safe state machines, 20 errata documents, lifecycle tests. It's the "defensive engineering" pass. But this came at the cost of more code, more complexity, and more documented workarounds.
  - v9 is the most mature: leaner production code, substantially more tests, cleaner module structure, fewer-but-better errata. It demonstrates the "write less, test more" philosophy. The agent-fox configuration reflects this too — higher budget ($15 vs $8), longer session timeouts — suggesting the operators learned
  that agent-fox needs more room to produce better results.

  The evolution from v7->v8->v9 shows a pattern of initial implementation -> hardening -> refinement. Notably, v8 and v9 were both produced by agent-fox 3.1.0 (just with different configs), while v7 used 3.3.1 — so the improvement from v8 to v9 is largely attributable to configuration tuning (more budget, longer
  sessions) rather than tool version changes.
