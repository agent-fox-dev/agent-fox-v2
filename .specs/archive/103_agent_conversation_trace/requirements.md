# Requirements Document

## Introduction

This spec re-purposes the `--debug` CLI flag to produce a structured JSONL
conversation trace for each orchestrator run. The trace captures what agent-fox
sends to the model (system prompt, task prompt) and what comes back (assistant
messages, tool invocations), enabling post-hoc debugging of agent–model
interactions.

## Glossary

- **Agent trace**: A JSONL file recording the chronological conversation between
  agent-fox and the LLM for a single orchestrator run.
- **run_id**: A unique identifier for an orchestrator run, generated by
  `generate_run_id()` (format: `YYYYMMDD_HHMMSS_{hex}`).
- **node_id**: Identifier for a task-graph node (a single coding session target).
- **Canonical message**: One of `ToolUseMessage`, `AssistantMessage`, or
  `ResultMessage` as defined in `agent_fox.session.backends.protocol`.
- **Tool input truncation**: Shortening string values in a tool's input dict to
  a maximum character length, appending a `[truncated]` marker.
- **AgentTraceSink**: The new `SessionSink` implementation that writes
  conversation-level trace events.

## Requirements

### Requirement 1: Trace File Creation

**User Story:** As a developer debugging agent behavior, I want a JSONL trace
file created for each debug run, so that I can inspect the full conversation
after the fact.

#### Acceptance Criteria

[103-REQ-1.1] WHEN `--debug` is passed to the `code` command, THE system SHALL
create an `AgentTraceSink` and register it with the `SinkDispatcher`.

[103-REQ-1.2] WHEN the `AgentTraceSink` writes its first event, THE system SHALL
create the trace file at `.agent-fox/audit/agent_{run_id}.jsonl`, where
`run_id` is the orchestrator's current run ID.

[103-REQ-1.3] THE `AgentTraceSink` SHALL write each event as a single JSON
object on its own line (JSONL format), flushing after every write.

#### Edge Cases

[103-REQ-1.E1] IF the `.agent-fox/audit/` directory does not exist, THEN THE
system SHALL create it before writing the first event.

[103-REQ-1.E2] IF a write to the trace file fails (I/O error), THEN THE system
SHALL log a warning and continue the session without interruption.

### Requirement 2: Session Init Event

**User Story:** As a developer, I want the trace to record what was sent to
the model at the start of each session, so that I can verify prompt
construction.

#### Acceptance Criteria

[103-REQ-2.1] WHEN a coding session begins execution (before the backend
message stream starts), THE system SHALL emit a `session.init` trace event
containing: `run_id`, `node_id`, `model_id`, `archetype`, `system_prompt`,
and `task_prompt`.

[103-REQ-2.2] THE `session.init` event SHALL capture the system prompt and task
prompt verbatim (no truncation).

### Requirement 3: Assistant Message Events

**User Story:** As a developer, I want every assistant response recorded in
the trace, so that I can see the model's reasoning and output.

#### Acceptance Criteria

[103-REQ-3.1] WHEN the session message stream yields an `AssistantMessage`,
THE system SHALL emit an `assistant.message` trace event containing: `run_id`,
`node_id`, `timestamp`, and the full message `content`.

[103-REQ-3.2] THE `assistant.message` event SHALL capture the content verbatim,
including any `[thinking]` prefix from extended thinking blocks.

### Requirement 4: Tool Use Events

**User Story:** As a developer, I want tool invocations recorded with their
arguments, so that I can see what the agent asked tools to do.

#### Acceptance Criteria

[103-REQ-4.1] WHEN the session message stream yields a `ToolUseMessage`, THE
system SHALL emit a `tool.use` trace event containing: `run_id`, `node_id`,
`timestamp`, `tool_name`, and `tool_input`.

[103-REQ-4.2] THE system SHALL truncate each string value in `tool_input` to a
maximum of 10,000 characters, appending `" [truncated]"` when truncation
occurs, AND return the truncated dict to the caller for inclusion in the event.

[103-REQ-4.3] THE system SHALL serialize non-string values in `tool_input`
as-is (no truncation).

#### Edge Cases

[103-REQ-4.E1] IF `tool_input` is empty or contains no string values, THEN THE
system SHALL emit the event with the input dict unchanged.

### Requirement 5: Tool Error Attribution

**User Story:** As a developer, I want tool errors recorded in the trace, so
that I can correlate failures with the tool that caused them.

#### Acceptance Criteria

[103-REQ-5.1] WHEN a session ends with `is_error=True` and a tool was the last
invocation, THE system SHALL emit a `tool.error` trace event containing:
`run_id`, `node_id`, `timestamp`, `tool_name`, and `error_message`.

### Requirement 6: Session Result Event

**User Story:** As a developer, I want the terminal outcome of each session
recorded, so that I can see token usage, duration, and status.

#### Acceptance Criteria

[103-REQ-6.1] WHEN the session message stream yields a `ResultMessage`, THE
system SHALL emit a `session.result` trace event containing: `run_id`,
`node_id`, `timestamp`, `status`, `input_tokens`, `output_tokens`,
`cache_read_input_tokens`, `cache_creation_input_tokens`, `duration_ms`,
`is_error`, and `error_message`.

### Requirement 7: Replace Existing JsonlSink

**User Story:** As a maintainer, I want the old debug JSONL sink removed so
that there is a single, well-defined debug trace format.

#### Acceptance Criteria

[103-REQ-7.1] THE system SHALL remove the `JsonlSink` class and its module
(`agent_fox/knowledge/jsonl_sink.py`).

[103-REQ-7.2] WHEN `--debug` is active, THE system SHALL register the
`AgentTraceSink` instead of the removed `JsonlSink`.

[103-REQ-7.3] THE system SHALL NOT write debug trace files to
`.agent-fox/{timestamp}_{session_id}.jsonl` (the old location).

#### Edge Cases

[103-REQ-7.E1] IF any module imports `JsonlSink`, THEN THE system SHALL update
or remove those imports so that the codebase compiles without errors.

### Requirement 8: Retention

**User Story:** As a developer, I want trace files kept indefinitely so that I
can inspect old runs without worrying about automatic cleanup.

#### Acceptance Criteria

[103-REQ-8.1] THE system SHALL NOT apply automatic retention or cleanup to
agent trace files (`agent_*.jsonl`).

[103-REQ-8.2] THE existing audit retention logic (`enforce_audit_retention`)
SHALL NOT delete `agent_*.jsonl` files when pruning old audit runs.

#### Edge Cases

[103-REQ-8.E1] IF the audit directory contains both `audit_{run_id}.jsonl` and
`agent_{run_id}.jsonl` for a run being pruned, THEN THE system SHALL delete
only the `audit_{run_id}.jsonl` file and preserve the `agent_{run_id}.jsonl`
file.
