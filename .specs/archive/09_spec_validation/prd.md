# PRD: Specification Validation

**Source:** `.specs/prd.md` -- Section 5 "Specification Validation" (REQ-080
through REQ-085), Section 4 "Validate Specifications" workflow, Section 7
"Lint Report" output specification, Section 8 error handling.

## Overview

The specification validation system checks spec files for structural and quality
problems before execution. It reports findings at three severity levels (Error,
Warning, Hint) and supports both static rule-based analysis and optional
AI-powered semantic analysis. Results are displayed as a Rich table or exported
as JSON/YAML.

This command is designed to be run before `agent-fox plan` or `agent-fox code`
to catch spec problems early, before they waste coding sessions and budget.

## Problem Statement

Specifications authored by humans (or generated by AI companions) may contain
structural defects: missing files, oversized task groups, broken cross-spec
dependency references, requirements without acceptance criteria, requirements
not traced to tests, or task groups without verification steps. Additionally,
acceptance criteria may be vague, unmeasurable, or describe implementation
rather than behavior. These problems lead to wasted coding sessions, incorrect
implementations, and costly rework.

A pre-execution lint pass catches these issues cheaply, saving the developer
both time and API cost.

## Goals

- Discover all spec folders and validate each against a catalog of static rules
- Report findings with severity (Error, Warning, Hint) and precise location
- Support table (Rich), JSON, and YAML output formats
- Optionally enable AI-powered semantic analysis of acceptance criteria
- Exit with non-zero code when any Error-severity findings exist
- Expose via `agent-fox lint-spec` CLI command

## Non-Goals

- Auto-fixing spec problems -- this command only reports, never modifies
- Validating the correctness of spec content beyond structural and semantic
  checks (e.g., whether a design is architecturally sound)
- Dependency graph construction or topological sort -- that is spec 02
- Running the plan command -- this is a standalone lint pass

## Key Decisions

- **Rules are pure functions.** Each static validation rule is a function that
  takes parsed spec data and returns a list of `Finding` objects. This makes
  rules independently testable and easy to add.
- **AI validation is optional and separate.** AI analysis uses the STANDARD
  model tier and is gated behind an `--ai` flag. It runs independently from
  static checks so the command works without API credentials.
- **Expected files per spec folder.** The validator expects five files:
  `prd.md`, `requirements.md`, `design.md`, `test_spec.md`, `tasks.md`.
  Missing files produce Error-severity findings.
- **Finding severity levels.** Error = will break execution or indicates a
  missing contract. Warning = likely to cause problems in practice. Hint =
  suggestion for improvement.
- **Exit code semantics.** Non-zero exit if any Error-severity findings exist,
  enabling CI integration.
